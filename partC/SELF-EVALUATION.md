# Self-Evaluation - Part C Checkpoint 4

## 1. Хэрэв шалгалтанд өөрөө орвол энэ кодыг өөрөө бичиж чадах уу?

Тийм, гэхдээ ai-ийн тусламжгүй шууд бичихэд хэцүү байх байсан. URL shortener-ийн үндсэн санаа болох урт URL хадгалах, богино code үүсгэх, code-оор redirect хийх хэсгийг би өөрөө ойлгож бичиж чадна. Express route үүсгэх, `POST /api/links`, `GET /api/links`, `GET /:shortCode` endpoint хийх, SQLite-д record хадгалах зэрэг үндсэн хэсгийг өөрөө хийх боломжтой.

Гэхдээ энэ project дээр зөвхөн "ажилладаг demo" биш, харин шалгуур хангахуйц implementation хэрэгтэй байсан. Тиймээс edge case, test coverage, OpenAPI documentation, AI session log, ADR, self-evaluation зэрэг олон давхар шаардлага гарсан. Энэ хэсэгт AI тусалсан нь ажлыг илүү цэгцтэй болгосон. Жишээ нь expired link дээр `410 Gone` буцаах, duplicate custom code дээр `409 Conflict` буцаах, custom code-д зөвшөөрөх тэмдэгтийг хязгаарлах, service/repository layer-ийг салгах зэрэг шийдвэрүүдийг би ойлгож байгаа ч бүгдийг эхнээс нь ганцаараа богино хугацаанд бүрэн бодож жагсаахад төвөгтэй байх байсан.

Өөрөөр хэлбэл би энэ project-ийн кодыг уншиж тайлбарлаж чадна, route-service-repository flow-ийг ойлгож байна, test яагаад pass болж байгааг мэдэж байна. Харин AI-гүйгээр хийвэл илүү удаан, магадгүй эхний хувилбар дээр test болон documentation дутуу гарах байсан. Тиймээс "өөрөө огт хийж чадахгүй" биш, харин "AI тусламжтайгаар илүү бүрэн, илүү шалгагдсан, илүү эмхтэй болгосон" гэж үнэлж байна.

## 2. Дахин хийвэл энэ ажлыг яаж өөрөөр хийх вэ?

 Part C-ийн AI usage тэмдэглэлийг project хийх явцдаа шууд өдөр бүр бичих байсан. Энэ удаа Part C дээр сүүлд нь тайлан болгож цэгцэлсэн. Сүүлд нь бичих боломжтой ч зарим AI prompt, шийдвэр, эргэлзээ яг ямар дарааллаар гарсныг мартуулсан. Дараагийн удаа AI session бүрийн дараа 5-10 мөрийн log шууд нэмбэл Reflect хэсэг илүү бодит, илүү нарийн болох байсан.

Дөрөвдүгээрт, frontend хэсгийг бага зэрэг засах, frontend дээр илүү ойлгомжтой харуулах, stats харах жижиг panel хийх, delete хийхээс өмнө confirm хийх зэрэг UX сайжруулалт нэмэх байсан.

## 3. Энэ туршлагаас юу сурсан бэ?

Энэ лабораторийн ажлаас хамгийн их сурсан зүйл бол AI-г зөв ашиглана гэдэг нь "орондоо хийлгэх" биш, харин "хамт бодож, дараа нь шалгах" процесс юм. AI хурдан санал гаргаж чаддаг ч тэр санал болгон project-д тохирно гэсэн үг биш. Жишээ нь endpoint-ийн нэр, database driver-ийн return утга, validation pattern, dependency сонголт зэрэг дээр заавал code уншиж, test ажиллуулж, project-ийн context-той тулгах хэрэгтэй.

Мөн test бичихийн ач холбогдлыг илүү сайн ойлгосон. Хэрэв test байхгүй байсан бол custom code feature ажиллаж байгаа мэт харагдах ч duplicate code, invalid custom code, expired link, click count зэрэг edge case буруу байж болох байсан. Харин `24` test pass болж байгаа нь implementation-ийн үндсэн behavior-ийг илүү итгэлтэй болгосон. Ялангуяа service test болон route test-ийг тусад нь бичсэн нь аль layer дээр алдаа байгааг ялгахад хэрэгтэй.

Эцэст нь AI ашигласан тухайгаа үнэнээр бичих нь ичих зүйл биш гэдгийг ойлгосон. Энэ даалгаврын гол санаа нь AI ашиглахгүй мэт дүр эсгэх биш, харин AI-г хэрхэн ашигласан, ямар хэсгийг өөрөө ойлгож баталгаажуулсан, ямар эрсдэл гарч болохыг тайлбарлах байсан. Миний хувьд AI хамгийн их тус болсон хэсэг нь edge case бодох, test coverage нэмэх, documentation цэгцлэх, architectural trade-off бодох байсан. Харин эцсийн баталгаажуулалт нь test ажиллуулах, code унших, шаардлагын checklist-тэй тулгах замаар хийгдсэн.
