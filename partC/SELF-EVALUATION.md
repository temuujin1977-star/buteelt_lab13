# Self-Evaluation

## 1. Энэ кодыг би өөрөө бичиж чадах уу?

Тийм, үндсэн санааг нь бол өөрөө бичиж чадна гэж бодож байна. URL shortener гэдэг нь урт URL аваад богино code үүсгээд, тэр code-оор буцаад original URL руу redirect хийдэг project. Энэ гол logic-ийг би ойлгож байгаа.

Express дээр route үүсгэх, `POST /api/links` endpoint хийх, SQLite database-д link хадгалах, `GET /:shortCode` дээр redirect хийх зэргийг өөрөө хийж чадна. Гэхдээ энэ project дээр зөвхөн ажилладаг demo биш, test, documentation, AI report, ADR гээд олон зүйл хамт хийх шаардлагатай байсан. Тэр бүхнийг нэг дор ганцаараа хийхэд удаан байх байсан.

AI хамгийн их тус болсон хэсэг нь edge case бодох байсан. Жишээ нь би эхэндээ valid URL оруулаад link үүсэж байна уу гэдгийг л бодох байсан байх. Харин AI-аас асуухад invalid URL, expired link, duplicate custom code, unknown short code, click count гэх мэт олон case нэмэх хэрэгтэйг ойлгосон.

Тиймээс би энэ project-ийг огт ойлгохгүйгээр AI-аар хийлгэсэн гэж бодохгүй байна. Би code-оо уншиж чадна, test юу шалгаж байгааг ойлгож байна. Гэхдээ AI тусалсан учраас илүү хурдан, илүү бүрэн болсон.

## 2. Дахин хийвэл яаж өөрөөр хийх вэ?

Дахин хийвэл checkpoint бүрийн өмнө жижиг checklist бичиж авах байсан. Жишээ нь:

- Энэ checkpoint дээр ямар feature нэмэх вэ?
- Ямар test нэмэх вэ?
- Ямар documentation өөрчлөх вэ?
- Commit message юу байх вэ?

Ингэвэл дараа нь commit history илүү цэгцтэй харагдана. Энэ удаа checkpoint-оор явсан нь зөв байсан, гэхдээ зарим үед яг юу нэмэхээ явцын дунд шийдсэн.

Мөн commit message-үүдийг эхнээс нь илүү адилхан style-тай бичих байсан. Жишээ нь `feat:`, `test:`, `docs:` гэх мэт prefix ашиглавал илүү цэвэр харагдана. Одоо зарим commit Монгол өгүүлбэр хэлбэртэй байгаа. Ойлгомжтой ч conventional commit талаасаа арай сул.

AI session log-ийг бас шууд явцдаа илүү сайн бичих байсан. Сүүлд нь бичихээр зарим үед AI-аас яг юу асуусан, ямар хариу авсан гэдгээ мартдаг. Тиймээс дараагийн удаа AI ашигласан бүрдээ 3-5 мөр тэмдэглэл хийгээд явах нь зөв юм байна.

Frontend хэсгийг бас арай гоё болгож болох байсан. Одоо ажиллаж байгаа ч энгийн. Цаг байсан бол stats харах жижиг хэсэг, delete хийхээс өмнө confirm гаргах, error message-ийг илүү ойлгомжтой харуулах байсан.

## 3. Энэ туршлагаас юу сурсан бэ?

Энэ даалгавраас хамгийн их сурсан зүйл бол AI-г ашиглахдаа заавал шалгах хэрэгтэй гэдэг. AI хурдан хариу өгдөг, гэхдээ тэр нь дандаа зөв гэсэн үг биш. Заримдаа project-д тохирохгүй endpoint нэр санал болгож болно. Заримдаа package-ийн behavior-ийг буруу тааж болно. Тиймээс "verify, don't trust" гэдэг санаа зөв юм байна.

Мөн test бичихийн хэрэгтэйг ойлгосон. Хэрэв test байхгүй байсан бол код ажиллаж байгаа юм шиг харагдах ч жижиг алдаа нуугдаж үлдэх байсан. Одоо 24 test pass болж байгаа болохоор үндсэн feature-үүддээ илүү итгэлтэй байна.

Route, service, repository гэж салгах нь бас хэрэгтэй санагдсан. Эхэндээ жижиг project дээр олон файлтай болгох шаардлагагүй юм шиг байсан. Гэхдээ feature нэмэгдэх тусам энэ бүтэц хэрэг болсон. Route нь request/response хариуцна, service нь logic хариуцна, repository нь database хариуцна гэдэг нь ойлгомжтой болсон.

Security талаас хэрэглэгчийн input-д шууд итгэж болохгүй гэдгийг сурсан. Custom code дээр ямар ч текст зөвшөөрвөл дараа нь асуудал болж магадгүй. Тиймээс зөвшөөрөх тэмдэгтийг хязгаарлах, duplicate code шалгах зэрэг нь жижиг project дээр ч хэрэгтэй.

AI ашигласан тухайгаа нуух биш, үнэнээр бичих хэрэгтэй гэдгийг бас ойлгосон. Энэ даалгаврын гол зорилго нь AI ашиглахгүй мэт жүжиглэх биш, харин AI-г яаж ашигласан, юу дээр тус болсон, ямар эрсдэлтэй байсныг тайлбарлах байсан гэж ойлголоо.

Миний хувьд AI туслаагүй бол project-ийг хийж болох байсан ч илүү удаан, магадгүй test болон documentation дутуу болох байсан. Харин AI тусалснаар би edge case-үүдийг илүү сайн харсан, report болон docs-оо илүү хурдан цэгцэлсэн. Гэхдээ эцэст нь test ажиллуулах, code унших, өөрийн үгээр тайлбарлах нь миний хийх ёстой ажил хэвээр байсан.
