# AI Usage Report

## Танилцуулга

Энэ тайлан дээр би Бие даалт 13 дээр AI-г яаж ашигласан талаараа бичлээ. Миний сонгосон сэдэв бол URL shortener. Гол санаа нь урт URL оруулахаар богино холбоос үүсгээд, тэр богино холбоосоор original URL руу redirect хийдэг жижиг web project юм.

Энэ даалгаварт AI ашиглаж болно, гэхдээ AI-аар хийлгэсэн зүйлээ нууж болохгүй гэж ойлгосон. Тиймээс би яг ямар хэсэг дээр AI-аас тусламж авсан, ямар хэсгийг өөрөө шалгасан, AI буруу санал өгөх боломж байсан эсэхийг энд бичиж байна. Миний хувьд AI-г бүх ажлыг шууд хийлгэх гэж биш, харин санаа авах, алдаа шалгуулах, test case бодох, бичсэн зүйлээ цэгцлэхэд ашигласан.

## 1. Юуг AI хийсэн, юуг өөрөө хийсэн бэ?

Part A дээр project-ийн санаа, stack сонголт, architecture төлөвлөх үед AI-аас асууж зөвлөгөө авсан. Жишээ нь URL shortener project-д ямар feature байх хэрэгтэй вэ, Node.js + Express + SQLite тохирох уу, directory structure-ийг яаж зохион байгуулах вэ гэх мэт зүйл асуусан. AI хэд хэдэн хувилбар санал болгосон. Тэр дундаас би өөрийн project-д хамгийн энгийн, даалгаврын хугацаанд багтах хувилбарыг сонгосон.

Part B дээр код бичих үед AI илүү их тус болсон. Гол API endpoint-уудыг яаж салгаж бичих, service layer дээр validation хийх, repository layer дээр database query бичих гэх мэт санааг AI-аас авсан. Гэхдээ AI-ийн бичсэн юмыг шууд сохроор хуулсан гэж хэлэхгүй. Repo дээр байгаа файлуудтай таарч байна уу, test pass болж байна уу гэдгийг шалгаж явсан.

AI-аас тусламж авсан гол зүйлс:

- URL shortener-д хэрэгтэй feature жагсаах.
- Express route, service, repository гэсэн бүтэцтэй болгох санаа авах.
- URL validation дээр ямар алдаа шалгахыг бодох.
- Expiration date, click count, unknown short code зэрэг edge case бодох.
- Custom short code нэмэхэд ямар validation хэрэгтэйг асуух.
- Unit test болон API test-д ямар case оруулахыг жагсаах.
- OpenAPI болон README-ийн эхний draft гаргах.
- Part C-ийн reflection дээр юу бичих ёстойг цэгцлэх.

Өөрөө хийсэн болон шалгасан зүйлс:

- AI-ийн санал project-д таарч байгаа эсэхийг уншсан.
- `npm.cmd test` ажиллуулж test pass эсэхийг шалгасан.
- Error status code-уудыг өөрөө ойлгож сонгосон.
- Custom code validation-д ямар тэмдэгт зөвшөөрөхийг шийдсэн.
- README, OpenAPI, AI session log-ийг project-ийн бодит байдалтай тааруулсан.
- Git дээр checkpoint бүрээр commit/push хийх workflow барьсан.

## 2. Part B дууссан эсэхийг шалгасан нь

Part B дээр дараах feature-үүд ажиллаж байгаа.

- Урт URL оруулаад богино URL үүсгэнэ.
- Үүссэн link-үүдийг жагсаана.
- Богино code-оор original URL руу redirect хийнэ.
- Redirect хийх бүрт click count нэмэгдэнэ.
- Хугацаа дууссан link дээр `410 Gone` буцаана.
- Буруу URL оруулбал `400 Bad Request` буцаана.
- Байхгүй short code дээр `404 Not Found` буцаана.
- Өөрийн custom short code оруулж болно.
- Давхардсан custom code оруулбал `409 Conflict` буцаана.
- Нэг link-ийн stats харж болно.
- Link устгаж болно.

Мөн даалгаврын шаардлагаар 10-аас дээш test байх ёстой. Манай project дээр нийт 24 test байгаа. Би дараах command ажиллуулж шалгасан.

```bash
npm.cmd test
```

Үр дүн нь 24/24 pass болсон. Энэ нь код 100 хувь төгс гэсэн үг биш, гэхдээ үндсэн feature болон edge case-үүд ажиллаж байгааг test-ээр баталсан гэсэн үг.

## 3. Hallucination жишээ 1

AI заримдаа ажиллаж болох ч project-д тохирохгүй санал өгч магадгүй. Нэг жишээ нь stats endpoint-ийн нэр. AI ерөнхийдөө `/api/stats/:shortCode` гэх мэт endpoint санал болгож болох байсан. Энэ endpoint ажиллаж болох ч манай project дээр link-тэй холбоотой бүх API `/api/links` доор байсан. Тиймээс би stats endpoint-ийг `GET /api/links/:shortCode` гэж хийсэн.

Хэрэв AI-ийн саналыг шууд дагаад өөр route хийсэн бол project-ийн бүтэц жаахан замбараагүй болох байсан. README дээр нэг өөр, OpenAPI дээр нэг өөр, test дээр бас өөр нэртэй болох эрсдэлтэй. Тиймээс AI-ийн санал зөв юм шиг байсан ч repo-ийн одоогийн бүтэцтэй харьцуулж шалгах хэрэгтэй гэдгийг ойлгосон.

Эндээс сурсан зүйл бол AI-ийн хариулт syntax-ийн хувьд зөв байлаа гээд project-ийн хувьд зөв гэсэн үг биш. Ялангуяа endpoint нэр, folder structure, naming style зэрэг нь тухайн project-ийн өмнөх хэв маягтай таарах ёстой.

## 4. Hallucination жишээ 2

AI package-ийн behavior-ийг заримдаа буруу тааж магадгүй. Жишээ нь SQLite дээр delete хийхэд устсан мөрийн тоог шалгах хэрэгтэй болсон. Зарим library дээр `rowCount` гэж байдаг. Гэхдээ энэ project дээр ашиглаж байгаа `sqlite` package-ийн `db.run()` result дээр `changes` гэж property ашиглаж байна.

Хэрэв AI-ийн таамгаар `rowCount` гэж бичсэн бол code зөв харагдаж болох ч delete logic буруу ажиллах байсан. Тиймээс repository file-ийн өмнөх pattern-ийг харж, `result.changes > 0` гэж шалгасан. Дараа нь test нэмсэн. Link үүсгээд устгахад үнэхээр байхгүй болж байна уу гэдгийг шалгасан.

Энэ жишээ надад AI library бүрийн жижиг detail-ийг яг зөв мэднэ гэж найдаж болохгүй гэдгийг харуулсан. Ялангуяа database, package, framework-ийн return value зэрэг дээр official documentation эсвэл project-ийн өмнөх code pattern-ийг харах хэрэгтэй.

## 5. Security/license анхаарсан жишээ

Checkpoint 3 дээр custom short code нэмсэн. Энэ үед хэрэглэгч өөрөө code бичиж өгч байгаа учраас validation хийх хэрэгтэй болсон. Хэрэв ямар ч шалгалт хийхгүй бол хүн `../admin`, `api/links`, хоосон string, зайтай үг, эсвэл маш урт string оруулж болно. Одоо шууд том security attack болохгүй байж магадгүй ч routing болон дараа нь project томрох үед асуудал болох боломжтой.

Тиймээс custom code дээр зөвхөн үсэг, тоо, `_`, `-` тэмдэгтийг зөвшөөрсөн. Мөн уртыг 3-32 тэмдэгтээр хязгаарласан. Энэ нь маш advanced security биш ч энэ жижиг лабораторийн project-д хангалттай гэж үзсэн.

Мөн duplicate custom code дээр анхаарсан. Хэрэв нэг хүн аль хэдийн байгаа code-ийг дахин ашиглавал өмнөх link солигдох эсвэл будилаан үүсэх эрсдэлтэй. Тиймээс давхардсан code дээр `409 Conflict` буцаадаг болгосон. Ингэснээр нэг short code нэг link-тэй л холбоотой байна.

License/package талаас бас нэг зүйл анхаарсан. AI заримдаа `nanoid` гэх мэт package ашигла гэж санал болгож магадгүй. Гэхдээ манай project дээр Node.js-ийн built-in `crypto.randomBytes` аль хэдийн short code үүсгэж байсан. Тиймээс шинэ package нэмээгүй. Шинэ package нэмбэл package-lock өөрчлөгдөнө, license шалгах хэрэг гарна, project шаардлагагүй томорно. Энэ удаа built-in боломж хангалттай байсан.

## 6. Юуг AI-аар хурдан хийсэн бэ?

AI хамгийн их хурд нэмсэн хэсэг бол эхний санаа гаргах, checklist гаргах, test case бодох байсан. Жишээ нь би өөрөө ганцаараа суувал "valid URL үүсэж байна уу" гэдгийг л хамгийн түрүүнд шалгах байсан. Харин AI-аас асуухад invalid URL, expired link, unknown code, duplicate custom code, click count гэх мэт олон case санаанд орсон.

Мөн documentation дээр AI тус болсон. README болон OpenAPI бичихэд endpoint бүрийн request, response, status code-ийг нэг бүрчлэн бичих хэрэгтэй. Энэ нь код бичихээс амархан юм шиг боловч жижиг алдаа их гардаг. AI draft гаргаж өгсөнөөр би түүнийг project-той тааруулж засахад илүү хурдан болсон.

Part C дээр ч AI report-ийн бүтэц гаргахад тус болсон. Ямар асуултад хариулах ёстой, hallucination/security жишээ яаж бичих вэ гэдгийг цэгцлэхэд хэрэг болсон.

## 7. Юуг AI-аар хийхэд удаан эсвэл төвөгтэй байсан бэ?

AI-аар хийхэд төвөгтэй байсан зүйл бол яг миний project-ийн context-ийг бүрэн ойлгуулах. Заримдаа AI ерөнхий зөвлөгөө өгдөг. Гэхдээ манай repo дээр аль хэдийн ямар файл байгаа, ямар test байгаа, ямар commit хийсэн гэдгийг сайн шалгахгүй бол хэт ерөнхий юм бичих магадлалтай.

Мөн AI-ийн бичсэн текст заримдаа хэт formal, хэт мэргэжлийн сонсогддог. Тэр нь багшид гоё харагдаж болох ч миний өөрийн бичсэн reflection шиг биш болчих эрсдэлтэй. Тиймээс энэ тайланг дараа нь илүү энгийн болгож зассан. Үүнийг би anti-pattern гэж харж байна: AI-аар бичүүлсэн текстийг ойлгохгүй байж шууд үлдээх нь буруу. Өөрөө ойлгож байгаа хэлээрээ засах хэрэгтэй.

Код дээр ч мөн адил. AI нэг дор том өөрчлөлт хийхээр санал болговол шалгахад хэцүү. Тиймээс checkpoint-оор бага багаар хийх нь илүү дээр байсан.

## 8. Skill atrophy буюу өөрөө сурахгүй болох эрсдэл

AI ашиглахад нэг эрсдэл нь өөрөө бодохоо болих. Хэрэв AI-ийн бичсэн code-г ойлгохгүйгээр хуулбал дараа нь шалгалт дээр тайлбарлаж чадахгүй. Тиймээс би дараах байдлаар энэ эрсдэлийг багасгах гэж оролдсон.

- AI-ийн гаргасан code-г уншсан.
- Яагаад ийм status code буцааж байгааг ойлгох гэж оролдсон.
- Test pass болохыг өөрөө ажиллуулж шалгасан.
- README болон report дээр өөрийн үгээр тайлбарласан.
- Хэт formal текстийг өөрийн ойлгодог хэлбэрт зассан.

Миний бодлоор AI-г бүр хорих биш, харин зөв ашиглах нь чухал. AI санаа хурдан өгнө, гэхдээ эцсийн ойлголт өөр дээр байх ёстой. Энэ project дээр би AI-аас тусламж авсан ч route, service, repository нь ямар үүрэгтэй, test юу шалгаж байгааг ойлгож байгаа.

## 9. Дүгнэлт

Энэ project дээр AI надад нэлээд тус болсон. Хэрэв AI байхгүй байсан бол би хийж чадах байсан ч илүү удаан, магадгүй test болон documentation дутуу гарах байсан. AI-аар хамгийн их туслуулсан зүйл бол edge case бодох, test case гаргах, documentation цэгцлэх, reflection бичих бүтэц гаргах байсан.

Гэхдээ AI-ийн хариултыг шууд итгэж болохгүй гэдгийг бас ойлгосон. Endpoint нэр, package behavior, security validation, dependency сонголт зэрэг дээр заавал өөрөө шалгах хэрэгтэй. Энэ даалгаврын гол санаа нь AI ашигласан эсэхээ нуух биш, харин AI-г яаж ашигласнаа үнэнээр бичиж, өөрөө юу ойлгосноо харуулах гэж ойлгосон.

Мөн энэ ажлаас нэг ойлгосон зүйл бол AI-аар хийлгэсэн кодыг өөрийн болгохын тулд заавал дахин унших хэрэгтэй юм байна. Эхэндээ AI-ийн өгсөн хариулт зөв юм шиг санагддаг. Гэхдээ яг өөрөө test ажиллуулаад, файл бүрийг нээгээд харвал зарим хэсгийг өөрөөр хийх хэрэгтэй болдог. Жишээ нь route нэрийг project-ийн өмнөх хэв маягтай тааруулах, README дээр бичсэн endpoint код дээр үнэхээр байгаа эсэхийг шалгах, custom code дээр ямар тэмдэгт зөвшөөрөхөө өөрөө шийдэх гэх мэт зүйлс байсан. Тиймээс би AI-г туслах багш шиг ашигласан гэж хэлж болно. Харин эцсийн хувилбарыг ойлгож, шалгаж, өөрийн үгээр тайлбарлах нь миний хариуцлага байсан.
